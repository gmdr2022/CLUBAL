VISÃO GERAL

O projeto evoluirá do modelo atual:

SAL – SESI Agenda Live
→ para uma solução independente e comercial:

CLUBAL – Club Agenda Live

Objetivo estratégico:

Transformar o sistema em um produto multiplataforma, independente de instituição, utilizável por:

• Clubes
• Academias
• Centros esportivos
• Condomínios
• Escolas
• Organizações em geral

O SESI será tratado como cliente, não como identidade central do software.

As implementações abaixo estão organizadas por horizonte de tempo, impacto técnico e viabilidade realista considerando:

• Desenvolvimento solo
• Uso principal de Python + Tkinter
• Sem infraestrutura paga inicial
• Evolução progressiva

ARQUITETURA DE DIRETÓRIOS (PADRÃO FUTURO – APPDATA)

O software passará a utilizar estrutura própria em:

%LOCALAPPDATA%\ClubAL\

Subpastas padrão:

%LOCALAPPDATA%\ClubAL\data\
→ Arquivos operacionais internos
→ Bases locais do sistema
→ Configurações e preferências

%LOCALAPPDATA%\ClubAL\logs\
→ Logs técnicos automáticos
→ Histórico de execução
→ Diagnóstico para suporte

%LOCALAPPDATA%\ClubAL\cache\
→ Cache de clima
→ Dados temporários
→ Arquivos reutilizáveis

%LOCALAPPDATA%\ClubAL\license\
→ Dados locais de ativação
→ Identificação da instalação
→ Estado da licença

Comportamento esperado do sistema:

Na inicialização, o CLUBAL deve:

• Garantir que todas as pastas existam
• Recriar automaticamente caso tenham sido apagadas
• Nunca depender da pasta principal do programa para escrita
• Criar (se possível) um atalho visível na pasta principal apontando para:

%LOCALAPPDATA%\ClubAL\logs\

Objetivo:

Facilitar manutenção técnica e diagnóstico futuro.

IMPLEMENTAÇÕES DE MÉDIO PRAZO (SEGURANÇA E LICENCIAMENTO)

START.EXE COMO ÚNICO PONTO DE ENTRADA

Fluxo de inicialização:

start.exe
 → chama módulo de verificação
 → comunica com security.py
 → valida integridade
 → valida licença
 → se OK → inicia backend + frontend
 → se NÃO OK → bloqueia execução

SISTEMA DE LICENÇA DE USO

Modelo:

• Chave única por unidade
• Intransferível
• Ativação individual

Comportamento:

Ao abrir o sistema:

• Solicitar chave de licença

Sem chave:

• Modo demonstração por 7 minutos

Durante o modo demonstração:

• Mostrar “Modo teste” discretamente
• Contador regressivo visível

Após o tempo:

• Retornar automaticamente para tela de ativação

MÓDULO DE VERIFICAÇÃO DE INTEGRIDADE

Criar sistema de proteção com:

• Arquivo interno de referência
• Hashes de verificação
• Comparação estrutural

Objetivo:

Evitar adulteração do software para burlar:

• Licença
• Tempo de uso
• Proteções internas

Se falhar:

• Impedir execução
• Registrar erro detalhado nos logs

LICENÇA ONLINE (FASE INTERMEDIÁRIA)

Modelo futuro:

• licence.py valida chave online
• Cada chave ativa apenas 1 instalação

Fluxo:

Usuário insere chave

Sistema consulta servidor

Verifica se já foi usada

Se não foi:
• Ativa localmente
• Registra uso online

Se já foi usada:
• Bloqueia ativação
